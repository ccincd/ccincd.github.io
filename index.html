配置和启动Zookeeper

Zookeeper是一款开源，高效且可靠的分布式协调服务框架，可以作为Dubbo的注册中心来提供分布式目录服务。

可在Zookeeper官网下载并安装，如果以单机启动的话，只需要一个Zookeeper实例，而想要以集群的方式来启动的话则每台机器上均需要安装一个Zookeeper实例。学习阶段，可能不愿意去花太多钱去购买几台云服务器，幸好Zookeeper支持在单机启动多个实例来组成一个伪集群。伪集群的配置可参考下图：

在我本机总共有三个Zookeeper实例，机器id分别为1、2、3，对应于上图中的server.1，server.2和server.3，由于三个实例都运行在本机，所以机器ip均为127.0.0.1。形如2888:3888的端口是Zookeeper集群之间交互的端口，第一个端口用于Follower和Leader之间的数据同步和其它通信，第二个端口用于Leader选举过程中投票通信，更详细的配置可参照ZooKeeper配置详解。

客户端与Zookeeper集群之间交互的端口是由clientPort来指定的，实例位于不同机器上的时候，端口可均为2181，而使用伪集群的时候则需要区分端口，比如server.1的配置文件使用端口2181，server.2使用2182（端口可以自己定义）等。

Linux调整/root硬盘空间

发表于 2016-08-21   |   分类于 技术   |  

最近在往Gitlab私服上Push代码的时候提示磁盘空间已满，首先想到的是去清理Tomcat的日志，于是写了个脚本把日志清空了，但/root下磁盘占有率依然很高，/home下还有很多空间剩余，于是就想是不是可以把/home下的空间挪动一部分到/root下，看了博客后发现方案可行，操作下，成功了，便想着把操作的步骤记下来，方便以后查看，也巩固一些运维的命令

系统情况

CentOS 6.5 64位

查看磁盘占用情况命令

  1. df -h 查看整个磁盘还剩多少空间
  2. du –max-depth=1 -h 当前文件夹磁盘使用情况
  3. du -sh 当前文件夹大小
  4. du -sh /root 某个路径下（root）的大小

四个命令都很实用，在开发机上执行命令一的时候提示 /dev/mapper/vg_localserver_lv_root （不同的系统、版本或硬件配置可能名称会不同）的占有率已经达到了100%，而 /dev/mapper/vg_localserver_lv_home 仅有个位数的占用率，调整的目标就是在不停掉进程的情况下从后者挪几百个G到前者的空间

Tips: 使用命令二可以快速定位硬盘上哪个文件夹占用了较大的空间，从而可以有针对性的进行清除，比如我定位到机器上 /usr/local/sonatype-work/nexus/storage/snapshots/ 占据了大部分的空间，发现原来是每次打过的war包没有定期删除，久而久之越来越多，把硬盘打爆的原来是它，而不是日志

操作步骤

a. 使用unmount命令卸载 /dev/mapper/vg_localserver_lv_home

这一步并没有卸载成功，提示有进程占用文件系统，这时先执行 fuser -m /home 命令，查找到占用进程的pid，然后执行kill -9 pid命令强制杀掉。这种方式比较暴力，最好确认下相应的pid可以kill掉，不然还是建议使用正常方式关闭进程

执行完成之后再次运行 df -h 命令发现，/dev/mapper/vg_localserver_lv_home 已经不显现了

b. 调整 /dev/mapper/vg_localserver_lv_home 的大小

如果不执行 e2fsck 命令则可能会提示不能跳过，应当先使用该命令来检查ext2、ext3、ext4等文件系统的正确性；第二条命令可以将目标分区调整（缩小）到期望的大小，这样就会有一部分空间（/home当前空间减去期望的大小）空余出来，可以分配给别的分区

由于步骤a执行了卸载，所以调整了空间之后还要执行挂载命令。执行完成之后再次运行df -h命令，可以发现 /dev/mapper/vg_localserver_lv_home 又回来了

c. 调整 /dev/mapper/vg_localserver_lv_home 的逻辑分区大小

前面已经调整过分区大小，这条命令的作用是调整逻辑分区的大小，这样空余出来的空间才可以真正被分配到其它的分区

执行完命令后可以使用 vgdisplay 命令查看目前空余磁盘空间的信息

d. 调整（增加） /dev/mapper/vg_localserver_lv_root 的大小

我有617G的空余空间，只增加了615G，留了一些空余，目的是为了避免出现Free PE空间不足的情况，具体留多少比较合适并没有考证

第一条命令增加逻辑分区的大小，第二条命令执行空间增加操作 -p 命令是为了打印执行进度，但在我操作的时候一点进度也没有打印出来，调整空间有点大，时间耗的有点久（几十分钟），导致我有些紧张，以为危险操作要失败了，后来想起来可以使用 df -h 命令查看分区的空间大小，之后看到分区空间正在稳步增加，于是就放心了。

Redis“漏洞”，SSH免登录攻击

发表于 2016-08-18   |   分类于 技术   |  

在没有安全防范的情况下使用Redis“漏洞”可以将本机SSH的id_rsa.pub写入到目标机器/root/.ssh/authorize_keys文件中，从而获取目标机器的免登录root权限，而这对于目标机器来说，无疑是一种灾难

被攻击

阿里云使用Redis的默认配置启动服务，没有设置密码，为了方便远程连接，服务器防火墙开了Redis默认的6379端口以及SSH的22端口，并且没有限制访问IP。这样做相当于把Redis暴露到公网，攻击者可以查询、修改Redis数据，甚至关闭、破坏Redis服务，考虑到目前Redis存储从数据量和重要性上需求都不大，所以并没有引起我对安全的重视。

最近发现之前配置的SSH免登录失效了，/root/.ssh/authorized_keys文件里面出现了类似于乱码的内容，如图1所示：

一开始并没有注意到文件开头的REDIS0006字样，以为是乱码，于是把内容清空了，并将自己的id_rsa.pub重新放入其中，但没过多久，文件又恢复了原样，并且有服务器被异地登录的异常。我试着搜了下REDIS0006，因此了解到这并不是简简单单的猜测到或者暴力破解到root密码的问题。

漏洞介绍

Redis服务暴露到公网，没有设置密码和其它安全策略，并且使用root权限启动，那么攻击者可以使用config命令将redis的dbfile设置为/root/.ssh下的authorized_keys，虽然名字与Linux系统的授权文件相同，但实际上是偷梁换柱，将Redis的数据库文件设置到.ssh文件夹下并覆盖原授权文件。

因为攻击者拥有Redis的读写权限，因而可以将自己本机的id_rsa.pub伪装成Redis值的方式写入到authorized_keys文件中，从而实现SSH免登录，并且由于授权文件位于/root下，因而获得了root权限。

  1. 本地生成SSH公钥和私钥：ssh-keygen -t rsa -C “yourname@youremail.com”

  2. 依次执行命令：

解决方案

以下解决方案并非要全部实现，但使用的越多，系统会越安全

  1. bind IP地址为127.0.0.1，而非使用默认的0.0.0.0
  2. 设置Redis密码
  3. 将config命令重命名为一串不易猜到的内容，如rename-command config “d901das8dhwq98sadwq9”
  4. 不使用root启动Redis服务
  5. 不使用默认的6379端口，或在防火墙中屏蔽掉Redis的对外接口，或配置IP访问规则，或者直接禁用掉SSH使用的22端口

攻击演练

在ZoomEye上搜port:6379，找到宿主机运行Linux系统并且没有设置密码校验的机器IP，按照步骤3的攻击方式，手动（真正的黑客一般不会这么玩）进行演练，约一个小时的时间才成功登录一台机器，虽然效率比较低，但获取那台机器的root权限之后我又以root身份登录了MySQL数据库，并且破解了Web应用的管理员账号，如果想要摧毁服务和系统，不费吹灰之力，当然我肯定不会这么做，因为我不是一个破坏者。

从上述演练可以看出来，被获取root权限的结果是多么的可怕，😱，身为一名开发人员，也应当有一定的安全意识，不然如果遇到互联网世界里的“坏人”，想要挽回损失就追悔莫及了。

好的编码习惯

发表于 2016-08-09   |   分类于 技术   |  

我写过不少烂代码，比仅仅实现功能强一点，花到重构和思考上的时间少之又少。最近有幸有一定的时间来重构项目代码，其中涉及到不少别人写的代码，整个过程还是有些让人头痛，但熬过去之后还是有不少感悟的

1、从注释说起

不写注释不重构，自己写的代码一个月后再看估计都想骂自己，当时不少的“灵光乍现”现在看起来却有点扎眼，原因是自己不得不动脑去思考那个时候为什么会那么写。

注释是最起码的要求，一段时间后，肯定会觉得当时自己那么啰嗦真是太有用了！

2、“强迫症”

有人说这是代码洁癖，不能容忍别人的风格和自己有悖。有人比较坚持，应当也有人比较排斥，工作一年多一些，我见过代码写的很随意的，也见过比较讲究的，而我个人确实有点强迫症，而且越发明显，不只难以忍受别人不写注释，也难以忍受乱加空格换行以及乱用中英文标点符号的，最忍受不了的当然还是打错别字，不仅仅是中文注释，也包括变量命名。

如果不注意这些小的地方的话，等代码量膨胀之后，整个系统看起来就像一锅粥，一旦有个较大的需求改动，改起来也是相当痛苦的。

八十分很多人都能达到，九十分以上的人就少的多了，当然接近满分的人应当是凤毛麟角了。能力上大家相差无几，但软素质却拉开了不小的距离。

写的好的代码不仅读起来舒服，用起来也是得心应手，希望自己写的代码也能达到并保持这种水平。

3、规范和边界

想到哪里写到哪里，以及想写到哪里就写到哪里，关于这点，我只想说，自由不是想干什么就干什么，自律才会有自由。规范大都是看别人的代码以及从与别人的讨论，甚至是争论中汲取并养成习惯的。写个Demo可以不讲求规范，但把系统一点点做大，没有规范的话路只会越走越窄，团队合作起来肯定也不会和谐。

我不是天才，我猜大部分人都不是，也许只有天才才可以Hold住那么多系统中的随机因素。作为一个普通人，我必须严守规范，一砖一瓦才能筑起高楼大厦。

写C语言的，一个函数洋洋洒洒几千行，各种位移，各种宏定义和0x常量，既高效又没Bug的，吾等只有膜拜，请收下我的膝盖，跪下我就不起来了。成不了那样的人，要想有所建树，除了要遵守规范，还要注意边界，一个功能该做什么就做什么，一个模块该干什么就干什么，一个接口可以解决的问题，不要写四个类似的接口出来。

4、影响与被影响

滴水不成海，独木难成林。我们不仅被身边的人影响着，也在影响着身边的人。放低自己的排斥感，去接纳所有好的东西，因为客观的说，自己的问题还不少，别人提出来的建议，只要去挖，下面必定有闪光的东西。团队和参与感永远是最快最有效的锻炼和成长方式。

5、Keep

说点题外话，坚持的力量真的非常可怕，每天只看十页书，周末也是，一年的时间就能看超过三千页！暂时用不到就当作知识储备，看过的东西不会浪费，因为计算机是一张网，每一个结都会让这张网越来越密，越来越结实。回到正题，踩过的坑记的最深，好的代码也是从一堆烂代码里面长出来的，坚持，加油。

6、Ending

今天看到两张Mac的桌面壁纸，觉得挺好看的，主角是我最喜欢的德国牧羊犬，放上来中和一下上面那一堆生硬的文字。
